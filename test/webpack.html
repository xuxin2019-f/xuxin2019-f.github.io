# webpack

# 概念&原理

webpack 实际上是一个模块打包工具，在 webpack 中，万物皆模块，它默认只识别 js 和 json 格式的文件，但它从配置的入口文件开始，从入口文件里识别模块依赖，分析项目结构（由于 webpack2 开始支持 commonjs 和 es6 语法，所以不论模块依赖是用什么写的都可以自动分析并转换编译），利用 loader将浏览器不能识别的模块和扩展语言转换编译成浏览器可识别的格式，递归地构建一个依赖关系图，包含应用程序需要的每个模块，将这些模块打包成一个或多个 bundle。打包成最终的文件，最终的文件中的模块实现是基于 webpack 自己实现的 webpack_require，把定义的模块内容挂载到 module.exports 上所以最后打包的文件可以跑在浏览器中。

# 手写 loader

用于模块解析，将模块原内容按需求转换成新内容。webpack 中所有内容皆模块 module，但 webpack 默认只识别 js 和 json，其他模块需要在 module 下的 rules 配置 loader（比如 less-load 会把 less 语法转换成 css）

loader 执行规则：从右到左，从下到上

比如解析less
less-loader将less解析成css，css-loader解析css模块之间的关系，合并成一个css，style-loader将css-loader生成的东西以style挂载到页面的header中
module:{ rules:[ { test:/\.less$/,
use:['style-loader','css-loader','less-loader'] } ] }

其实 loader 的原理就是一个函数（注意因为要用 this，这里不能是箭头函数），参数就是要处理的模块，将模块处理好后再返回 举例：

//1.最简单版本：
// index.js
console.log('xx')
// loader.js
module.exports = function (source) {
  // 这里要用到this，所以需要声明式函数
  console.log(source, this.this.query)
  return source.replace('xx', 'req')
}

//2.配置参数：通过this.query获取
module.exports = function (source) {
  return source.replace('xx', this.query.name)
}
// 配置文件：
modules: {
  rules: [
    {
      test: /\.js$/,
      loader: path.resolve(__dirname, './loader.js'),
      options: {
        name: 'req',
      },
    },
  ]
}
//3.返回多个信息：this.callback
module.exports = function (source) {
  const result = source.replace('xx', this.query.name)
  this.callback(null, result)

  //this.callback = {
  //err:Error|null,
  //content:string|Buffer,
  //sourceMap? :SourceMap,
  //meta?:any
  //}
}
//4.处理异步事务:this.async
module.exports = function (source) {
  const callback = this.async()
  setTimeout(() => {
    const result = source.replace('xx', this.query.name)
    callback(null, result)
  }, 3000)
}

 module.exports = {
  entry:{
  // 每个入口都是一个chunk，每个入口名都是一个chunkname
    name1:'./src/name1.js',
    name2:'./src/name2.js'
  },
  output:{
    path:path.resolve(__dirname,'./dist')
  // name和chunkname对应， chunkhash与每一个chunk文件联系在一起，只有chunk文件改变，chunkhash才改变，如果不变，浏览器继续使用内容，起到优化的作用
    filename:'[name]_[chunkhash:8].js',
  }
}

# 代码分离

代码分割就是我们根据实际业务需求将代码进行分割，然后在合适的时候在将其加载进入文档中

代码分离 code spliting https://webpack.docschina.org/guides/code-splitting/

代码分割教程https://www.jianshu.com/p/2cc8457f1a10

代码分离将不同的代码分离到不同的 bundle 中（比如把第三方模块分离出去），控制资源加载优先级，获取更小的 bundle，极大提高加载速度

代码分离方式：

  • 入口起点配置：如果入口 chunk 之间包含一些重复的模块，那些重复模块都会被引入到各个 bundle 中
  • 使用 SplitChunksPlugin 插件（这个插件不需要任何额外下载，直接在 webpack.config.js 下配置 optimization 即可），即 webpack 笔记里写的，此插件将所有公共的模块都抽离出去，提取到已有的 entry chunk 或一个新生成的 chunk。

在这个插件的作用下：

webpack 将根据以下条件自动拆分代码块：

  • 会被共享的代码块或者 node_mudules 文件夹中的代码块
  • 体积大于 30KB 的代码块（在 gz 压缩前）
  • 按需加载代码块时的并行请求数量不超过 5 个
  • 加载初始页面时的并行请求数量不超过 3 个
optimization: {
     splitChunks: {
       chunks: 'all'
     }
   }

splitChunks 的一些特殊选项及其默认值：

miniSize:30000,最小尺寸，当模块大于 30kb

minChunks：1 打包生成的 chunk 文件最少有几个 chunk 引用了这个模块

maxAsyncRequests：5 动态加载请求（异步请求数）

maxInitialRequests：3 最大初始化请求数

chunks(选择分割哪些代码)：默认 async（按需加载的代码），all（全部代码），initial（页面初始化的代码）

以下是由社区提供，一些对于代码分离很有帮助的 plugin 和 loader：

  • mini-css-extract-plugin：用于将 CSS 从主应用程序中分离。
  • bundle-loader：用于分离代码和延迟加载生成的 bundle。
  • promise-loader：类似于 bundle-loader ，但是使用了 promise API。

# tree shaking

摇树将未被使用（即没有被 import按需引入的）的 exported member 标记为 unusage，并使它不再 export

注意副作用！！比如 css 等文件实际上是直接 import 导入，而不是按需导入的，需要额外配置

//package.json
//"sideEffects":false正常对所有模块进⾏tree shaking
// 改为对某些特定模块不进行摇树
"sideEffects":['*.css','*.less','*.png'，’@babel/ployfill]

sideEffects 为副作用，在什么都不设置的情况下，即使模块没有被按需导入，如果该模块里存在副作用代码，该部分代码仍会被保留下来，这样摇树后的项目体积并没有很好的减小，所以在可以全局摇树的情况下，可以直接设置'sideEffetcs':false

# 10.webpack-merge

见 webpack 笔记，分为两种方式

  • 下载引入 webpack-merge，分别建立 webpack.base.js,webpack.dev.js,webpack.pro.js,用 merge 分别在 dev 和 pro 的 js 文件夹中合并 base，然后在 package.json 中写 build 和 dev 两个脚本分别引用，注意：dev 开发模式需要先经过服务器，所以写法是

    "scripts":{
     "build": "webpack --config ./webpack.pro.js",
         // 注意：开发环境需要先经过服务器才行，所以需要写webpack-dev-server(模拟接口写在了server.js里)，这是原来server的配置
     "dev": "webpack-dev-server --config ./webpack.dev.js"
     // "server":'webpack-dev-server'
}

  • 在 base.js 中引入 dev.js 和 pro.js，在 scripts 脚本中，给生成环境加变量 env.production，根据这个变量在 base.js 判断是否存在，如果存在则 merge pro.js，负责合并 dev.js

 "scripts": {
    "test": "echo \"Error: no test specified\" && exit 1",
    "base_build": "webpack --env.production --config ./webpack.base.js",
    "base_dev": "webpack --config ./webpack.base.js"
  },

  // base.js
  module.exports = env => {
  if (env && env.production) {
    return merge(commonConfig, prodConfig);
  } else {
    return merge(commonConfig, devConfig);
  }
};
